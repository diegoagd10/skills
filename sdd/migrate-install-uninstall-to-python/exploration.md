## Exploration: migrate `ai-harness install` / `uninstall` to Python + Rich

### Current State
- The Go binary `cli/cmd/ai-harness/main.go` dispatches everything from `Run()` in `cli/cmd/ai-harness/run.go`.
- `install` and `uninstall` are handled by `runInstall()` (lines 137–225), which:
  1. Parses `--repo` and `--harness` flags.
  2. Builds a `install.Config` from `$HOME`.
  3. Calls `install.Install()` / `install.Uninstall()` in `cli/internal/install/install.go`.
  4. For OpenCode, also calls `commands.Generate()`, `opencode.Generate()`, and writes the manifest.
- `sdd-status` and `sdd-continue` have already been migrated to Python (`cli/src/ai_harness/cli.py`). The Go paths still exist as the reference/fallback, but the Python Typer app registers the same hyphenated command names and preserves the JSON contract.
- The Python project is uv-managed (`cli/pyproject.toml`, `uv.lock`, `Makefile`) with `typer` and `rich` as runtime deps and `pytest`/`ruff`/`coverage` as dev deps.

### Affected Areas
- `cli/cmd/ai-harness/run.go` — `runInstall`, `resolveSelection`, `promptHarnesses`, `homeConfig`, `syncOpencodeCommands`, `syncOpencodeConfig`, `formatOutcome` will shrink or move.
- `cli/internal/install/install.go` — all copy/remove/manifest logic must move out of Go.
- `cli/internal/commands/commands.go` — OpenCode slash-command generation currently lives in Go; needs a Python equivalent or a CLI call from Python to the remaining Go binary.
- `cli/internal/opencode/opencode.go` — `opencode.json` generation with `{{HOME}}` substitution must move to Python.
- `cli/src/ai_harness/cli.py` — add `install` and `uninstall` Typer commands.
- `cli/src/ai_harness/` — new module(s) needed for install/uninstall/file operations.
- `cli/tests/test_cli.py` — add Typer-level tests for the new commands.
- `e2e/e2e_test.sh` — currently builds the Go binary and runs `ai-harness install`; must be updated to exercise the Python entry point (or keep a compatibility path).
- `README.md` / `cli/README.md` — docs reference the Go CLI and `--harness` text prompt; update for Rich menu UX.

### Approaches
1. **Full Python port (recommended)**
   - Move `install.Install()`, `install.Uninstall()`, command generation, and `opencode.json` generation into Python modules under `cli/src/ai_harness/install/`.
   - Add `install` / `uninstall` Typer commands in `cli.py`.
   - Use Rich `ListPrompt` or a custom `rich.live`/`rich.console` checkbox list for the interactive harness selector (j/k/↑/↓, space, enter, escape).
   - The Go binary becomes a thin wrapper that delegates install/uninstall to the Python console script, or is retired later.
   - **Pros**: single source of truth; unlocks Rich UX; consistent with sdd-status migration; fully testable with pytest; no Go/Python interop at install time.
   - **Cons**: larger initial migration; must reproduce manifest format and safety validations exactly to stay compatible with existing installs.
   - **Effort**: High.

2. **Python frontend + Go backend**
   - Python owns the Rich menu and CLI surface, then shells out to the Go binary with `--harness a,b,c` to perform the actual copy/manifest work.
   - **Pros**: smaller Python change; reuses proven Go install/uninstall/manifest logic verbatim.
   - **Cons**: does NOT move copy/remove logic out of Go (violates the requirement); more process spawning; harder to test end-to-end in Python.
   - **Effort**: Medium.

3. **Go frontend with Rich-like rendering (not recommended)**
   - Keep install/uninstall in Go but replace the text prompt with a Go TUI library.
   - **Pros**: minimal file movement.
   - **Cons**: does not match the Python migration direction; adds a new Go dependency; Rich is a project requirement.
   - **Effort**: Medium.

### Recommendation
Adopt **Approach 1 (Full Python port)**. It satisfies the user requirement to move artifact copy/remove out of Go, aligns with the existing sdd-status/sdd-continue migration, and lets Rich own the human-facing menu. The Go install package can be deleted once parity tests pass.

### Risks
- **Manifest compatibility**: existing users have `~/.config/ai-harness/install-manifest.json` written by Go. The Python port must read/write the identical schema or users will be unable to uninstall previously installed files.
- **Path safety**: Go `install.Uninstall()` validates every manifest entry is inside managed roots and rejects directory/traversal entries. This safety logic must be reproduced exactly.
- **Copy semantics**: Go `copyDir()` rolls back partial copies on failure and preserves unmanifested user files. Python `shutil.copytree` behavior differs; need custom walk or careful option flags.
- **OpenCode commands + config**: these are currently generated by Go packages with placeholder substitution. Moving them requires parsing `prompts/commands/*.md` frontmatter and substituting `{{ORCHESTRATOR_AGENT}}`, `{{SKILLS_DIR}}`, `{{CWD_NOTE}}`, `{{ARGS}}`, plus `{{HOME}}` for `opencode.json`.
- **No TTY fallback**: if stdin is not a terminal, the menu must not block; default to all harnesses (matching current Go behavior) or require `--harness`.
- **Empty selection UX**: Rich menu must allow cancel/empty selection and print the requested "nothing selected" message.
- **Engram unavailable**: this exploration could not be persisted to Engram because `mem_save`/`mem_context` tools are not present in the current toolset.

### Ready for Proposal
**Yes**, with one clarification needed: should the Go install package be deleted as part of the migration, or kept as a fallback until the Python implementation is proven? The existing tests and E2E script currently depend on the Go path, so the cutover strategy should be specified in the proposal.
